Sticky NOTE: Details about SDR# 1778+ and this plug-in: (not a complete list)

- The Telerik UI that SDR# (1732+) now uses poses some problems for the (free) plug-in developers.
  - The Telerik controls are NOT free to use and so we need to use the standard/user controls the best we can.
    These controls are limited in what can be done to them to adapt them to be used with the Telerik UI to
     minimise the jarring differences.
    The progressbar colours cannot be changed.
    In dark themes, these limits become more noticeable.

NOTE: While I have been trying to maintain backwards compatibility with older SDR# versions, this is becoming
       more laborious. Specifically versions from 1717-1777 may not be displayed as intended.
      Testing the plug-in with effectively three different types of SDR# is now required. (pre-Telerik, Telerik with side panel, Telerik with dock control)

1.3.11.0 - Release                                                                        19th June 2026

Fixed: Race conditions in DiskWriterThread (SimpleRecorder)
- _diskWriterRunning and _diskWriterPause fields were not marked volatile.
  Without volatile, the CPU/JIT could cache the values and the recording thread would never
  see the stop signal from the UI thread, causing a hung thread on Stop.
  Both fields are now declared volatile.

Fixed: Potential use-after-free in DiskWriterThread drain loop (SimpleRecorder)
- After the main recording loop exits, the drain loop used _bufferPtr directly.
  A concurrent Dispose() call could free the underlying buffer while the drain loop
  was still reading from it. The pointer is now captured into a local variable with a
  null-check before the drain loop runs.

Fixed: WAV file header not written when stream stops due to full disk/size limit (SimpleRecorder / SimpleWavWriter)
- If DiskWriterThread exited because IsStreamFull became true, it would set
  _diskWriterRunning = false but the WAV header would only be updated when StopRecording()
  eventually called Flush(). If the application crashed in that window, the file was corrupt.
  DiskWriterThread now calls _wavWriter.Close() directly before signalling completion.
  SimpleWavWriter.Close() is now idempotent (returns instead of throwing when already closed).

Fixed: WAV file left open if Thread.Join() throws in StopRecording() (SimpleRecorder)
- StopRecording() now wraps the Join() call in try/finally to guarantee Flush() runs
  and the WAV file is properly closed even if an exception occurs.

Fixed: StereoToMono discarded right channel entirely (SimpleWavWriter)
- The conversion loop was writing data[num] (left channel only) instead of averaging
  both channels. PCM8Mono and PCM16Mono recordings lost all audio from the right channel.
  Fixed to: (data[num] + data[num + 1]) * 0.5f

Fixed: UpdateLength() called on every audio buffer write causing excessive disk seeks (SimpleWavWriter)
- WriteStream() called UpdateLength() for every audio block written, which performs
  two Seek() + two Write() operations per call. Under normal recording rates this causes
  significant unnecessary disk I/O. UpdateLength() is now called only in Close().

Fixed: Incorrect recording duration calculation for PCM16Stereo, Float32, PCM16Mono (AudioRecorderPanel)
- _writeLength was calculated as bytesWritten / _recordSamplerate, which did not account
  for bytes-per-sample (2 for PCM16, 4 for Float32). This caused:
    PCM16Stereo: duration 2x too high
    Float32:     duration 4x too high
    PCM16Mono:   duration 2x too high
  The calculation now uses the correct bytes-per-frame divisor for each format.
  The incorrect compensation (num3 /= 2) in the display timer was also removed.

Fixed: _fileIndexer not reset between recording sessions (AudioRecorderPanel)
- The conflict-avoidance file index counter was never reset when starting a new recording
  session. Files from subsequent sessions would continue numbering from the previous session.
  _fileIndexer is now reset to 0 each time recording starts.

Fixed: New file created on frequency change even when option was disabled (AudioRecorderPanel)
- The condition for creating a new file on frequency change was:
    _oldFrequency != _control.Frequency && (NewFileFrequencyEnable || SignalIsActive())
  This incorrectly triggered a new file whenever the frequency changed AND the squelch was
  open, regardless of the NewFileFrequencyEnable setting. Fixed to check only:
    _oldFrequency != _control.Frequency && NewFileFrequencyEnable

1.3.10.0 - Release

Fixed: 'Sample Format' '32 Bit IEEE Float Stereo'
- Recording with this Sample Format would see stuttering of audio because of an oversized buffer used for write buffer.
  Only half of this buffer was been written too, the other half was blank.
  I guess no-one ever uses '32 Bit IEEE Float Stereo' as it looks like it never worked going back to Vasili's original version.
